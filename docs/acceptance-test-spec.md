# 受け入れテスト仕様書案

## 1. 目的
営業アプローチリスト作成ツールについて、**テストが通る = 現場で使える**と言える受け入れテスト仕様を定義する。

初回受け入れでは、網羅性や高度な最適化よりも、以下を優先する。

- 誤った企業情報を出さない
- NGエリアを絶対に含めない
- 同名企業の誤紐づけを防ぐ
- 情報欠損があっても営業が使える形にする
- スコアリング結果に説明可能性がある
- コマンド単位で段階的に確認できる

## 2. 前提
- 正本要件は GitHub Issue #9
- `NGエリア = 契約済み市区町村`
- 対象地域は人口規模条件で指定できればよい
- usable 条件は `企業名 + 所在地 + (Webサイト or 電話番号)`
- usable 未達企業も最終CSVに残し、判別可能なマーキングを付与する
- 外部情報源は許容するが、最終CSVでソース明示が必要
- 問い合わせフォームURLは、公式サイト起点なら外部フォームでも可
- スコアリングは受け入れ対象に含める
- スコアリングは精度完成より、理由の説明可能性と設定調整可能性を優先する
- 今回は重複統合を受け入れ対象外とする

参照:
- [GitHub Issue #9](https://github.com/kosukekikuchi-coder/Approachlist/issues/9)

## 3. 受け入れ対象範囲
対象とする機能は以下。

- 対象地域の指定とNGエリア除外
- 経営者団体HP探索
- 会員企業名抽出
- 企業詳細情報収集
- usable 判定とマーキング
- 外部情報源のソース明示
- 問い合わせ先収集
- 営業優先順位スコアリング
- スコア設定の調整反映
- CSV出力
- ログ、進捗、品質レポート
- コマンド単位の段階実行

対象外は以下。

- 重複統合の品質評価
- 大規模件数での性能上限評価
- スコアの営業成果への実証評価

## 4. 現場で使えると判断する条件
以下を満たした場合、初回受け入れとして「現場投入可能な最小品質」と判断する。

1. 人口規模条件で対象地域を指定できる
2. 契約済み市区町村が最終成果物に含まれない
3. 同名企業など判別困難な場合、誤情報で埋めず空欄または低信頼で扱う
4. usable 判定ができ、未達レコードにもマーキングがある
5. usable 未達企業も除外されず確認可能である
6. 企業詳細情報の取得ソースが最終CSVで追跡できる
7. 問い合わせフォームURLの取得ルールが守られる
8. スコア、ランク、理由、信頼度が出力される
9. スコア理由を営業担当者が読んで理解できる
10. スコア設定変更が結果に反映される
11. 一部失敗や情報不足があっても全体停止しない
12. 最終CSVをExcelで文字化けなく開ける
13. 各コマンドを独立実行、再実行できる

## 5. テスト観点
### 5.1 対象地域指定
- 人口規模条件で対象地域を絞れる
- NGエリアが除外される
- 境界人口の地域を正しく扱える

### 5.2 団体HP探索
- 対象地域に対応する団体HPを取得できる
- 無関係サイトを誤採用しない
- 会員リスト有無を判定できる

### 5.3 会員企業抽出
- 団体HP構造差異があっても企業名抽出できる
- 一部失敗でも全体停止しない
- ノイズ混入を抑える

### 5.4 企業詳細情報収集
- 企業名、所在地、電話、Webサイト等を取得できる
- `mailto:` を抽出できる
- 公式サイト起点の外部フォームを取得できる
- 同名企業誤紐づけを避ける
- 取得不能時に無理に埋めない

### 5.5 usable 判定
- usable 条件を正しく判定できる
- 未達企業も残し、マーキングできる

### 5.6 ソース明示
- 外部情報源を含め取得元を追跡できる
- ソース未記録を防ぐ

### 5.7 スコアリング
- スコア、ランク、理由、信頼度を出力できる
- 情報不足でも処理継続できる
- 理由が説明可能である

### 5.8 スコア設定調整
- 配点や辞書変更が結果に反映される
- 不正設定を検知できる

### 5.9 CSV出力
- UTF-8 BOM付きで出力される
- 必要列が揃う
- Excelで実用的に開ける

### 5.10 ログ・進捗・品質レポート
- ステップ進捗が確認できる
- 品質指標が確認できる
- エラーがログに残る

### 5.11 コマンド単位実行
- 各ステップを独立実行できる
- 再実行しやすい
- 中間成果物で確認できる

## 7. 中・低優先度の追加候補
### 中優先度
- 条件一致地域が0件のとき正常終了できる
- 会員一覧非公開時に未取得理由を記録できる
- `mailto:` 抽出の正確性
- 公式サイト不明時に `email` と `contact_form_url` を無理に埋めない
- usable 未達企業でもスコア出力継続
- スコア理由の可読性
- 不正設定ファイルの検知
- レポート件数とCSV件数の整合
- Claude Code / Codex CLI で同等構造の出力になる

### 低優先度
- HTML構造差異への安定性
- 法人格表記ゆれへの対応
- 極端設定時のランク分布
- ログ文言の分かりやすさ
- 中間CSVの見やすさ
- 大量件数時の性能傾向

## 8. 推奨出力
受け入れ評価を低コストで安定させるため、以下を推奨する。

- 最終成果物: CSV
- 進捗レポート: CSV または JSON
- 品質指標レポート: CSV または JSON
- エラーログ: テキストログ
- 中間成果物: 各コマンドごとの CSV

画面表示は補助確認とし、正式判定はファイル出力ベースで行う。

## 9. 未確定論点

- 企業詳細ソース列の正式名と構造
- 団体ソースと企業詳細ソースを分けるか
- スコアランク閾値
- `score_confidence` の表現形式
- スコア理由の文字数上限
- 進捗・品質レポートの正式フォーマット
- CLI間差異をどこまで許容するか
- 将来重複統合を再度対象化する際の代表値ルール

## 10. 判定文
初回受け入れでは、次を満たせば「現場で使える」と判断する。

**人口条件で絞った対象地域について、NGエリアを含めず、誤企業情報を安易に付与せず、usable可否と情報ソースを明示した営業先CSVを、スコア理由つきでコマンド単位に確認しながら出力できること。**

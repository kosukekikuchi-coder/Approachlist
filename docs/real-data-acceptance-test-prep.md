# 実データ受け入れテスト準備リスト

## 目的
fixture ベースの確認から次へ進み、**本番でも使用する地域リスト / NG地域リスト** を使って小規模な受け入れテストを行うための準備事項を整理する。

## 現時点の認識
- 地域リストの候補として、以下のCSVがすでに存在する  
  [まいぷれパートナー開拓_地域貢献関心群 - 都市雇用圏（10～50万人）.csv](/C:/Users/fln_user/Documents/作業用フォルダー/まいぷれパートナー開拓_地域貢献関心群%20-%20都市雇用圏（10～50万人）.csv)
- このCSVには以下が同居している
  - OK/NG判定
  - 地方
  - 都市雇用圏名
  - 圏域人口
  - 区分
  - 構成自治体
- つまり、**本番で使う地域リストの元データ** としては使えそうだが、現行CLIの `config/areas.csv` / `config/contracted.csv` にそのまま入る形式ではない

## 最優先でやること
### 1. 正本を決める
- 地域リストの正本は、上記CSVを使う
- NG地域リストの正本も、同じCSVの `エリアOK/NG` 列を起点にする

### 2. テスト対象を小さく絞る
- 初回の実データ受け入れは、1〜3地域に絞る
- 条件:
  - `OK` 地域を少なくとも1件含める
  - `NG` 地域を少なくとも1件含める
  - できれば人口境界に近い地域も含める

### 3. CLI入力形式へ変換する
- `config/areas.csv` に必要な最小列
  - `municipality`
  - `population`
- `config/contracted.csv` に必要な最小列
  - `municipality`
- 今回の本番CSVからは、少なくとも以下の変換が必要
  - `都市雇用圏名` を対象地域名として使うか決める
  - `圏域人口` を数値化する
  - `エリアOK/NG` が `NG` のものを NG地域リストに落とす

## 確認すべき論点
### 地域名の単位
- 現行CLIは `municipality` 前提
- 手元の本番CSVは `都市雇用圏名（中心都市）` 前提
- そのため、次のどちらかを決める必要がある
  - 当面は `都市雇用圏名` を `municipality` 相当として扱う
  - CLI側を `municipality` ではなく `area_name` ベースへ寄せる

### NGの意味
- CSVの `エリアOK/NG` には複数表現がある
  - `NG（周辺ふくむ）`
  - `NG（近隣なら検討可）`
- 受け入れテスト上は、まず **どちらも NG として除外** する前提で進めるのが安全

### 構成自治体の扱い
- 現行CLIは構成自治体の分解に対応していない
- 初回実データ受け入れでは、まず **都市雇用圏単位で判定** する
- 構成自治体単位の除外は次段階に回す

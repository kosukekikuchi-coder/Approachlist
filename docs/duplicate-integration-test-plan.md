## 最初に確認したい観点

### 1. 同一企業を1件に統合できる
- 同じ企業が複数ソースに出ても、最終出力が重複行にならない
- `company_name + municipality` だけでなく、住所や電話番号も見て安全に統合できる

### 2. 異なる企業を誤って統合しない
- 似た会社名でも、住所や電話番号が一致しなければ別企業として残す
- 判断に迷う場合は統合しない

### 3. ソース情報を失わない
- 統合後も、どのソースから確認したか追跡できる
- 少なくとも代表ソースと補助ソースを残せる形が望ましい

### 4. 情報が強い方を採用する
- 公式サイトまたは詳細情報の強いソースがある場合は、そちらを優先できる
- 住所、電話番号、Webサイトなどが補完される

### 5. 表記ゆれに耐えられる
- `株式会社くまがい印刷`
- `くまがい印刷`
- `株式会社　フロム・エー`

のような表記差があっても、根拠が揃えば同一企業候補として扱える

## 初回の受け入れ条件案
次を満たしたら、重複統合の最初の一歩として合格とみなす。

1. `株式会社くまがい印刷` が複数ソースに出ても、最終出力では1件になる
2. 統合後も、住所・電話番号・Webサイトが欠落せず保持される
3. 統合に使ったソースを後から追跡できる
4. 判断に迷うケースでは無理に統合しない

## 実装前に決めたいこと
- 統合キーを何にするか
  - `company_name + municipality`
  - `company_name + phone`
  - `company_name + address`
  - 複合判定
- 統合後にソースを1つだけ残すか、複数保持するか
- 住所や電話番号が食い違った時の優先順位

## 現時点のおすすめ
初回は、以下のような保守的な方針が安全。

- `company_name` の近似だけでは統合しない
- `municipality` に加えて `phone` または `address` の一致を重視する
- 公式サイト由来の詳細を優先する
- 迷ったら別件のまま残す

## 次にやるべきこと
1. この観点をベースに、重複統合の受け入れテストケースを小さく1〜3件作る
2. そのテストを通す最小実装方針を決める
3. 秋田市のケースで先に試す
